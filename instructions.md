# LLM-Utils: Launch & Evaluation Instructions

This repository contains scripts and frameworks for launching `llama-server` instances and running automated performance/quality evaluations on GGUF models.

---

## 1. Launching Servers

### Single Server: `launch-server.sh`
Used for running a single model instance.
- **Location:** `utils/launch/launch-server.sh`
- **Usage:** `./utils/launch/launch-server.sh [model-type]`
- **Supported `model-type` args:**
  - `gemma-26b-q5` (Default)
  - `gemma-31b-iq4`
  - `gemma-31b-q4s`
  - `qwen-27b`

### Dual Server: `launch-dual.sh`
Used for running the "Combined Framework" (Main Model + Side/Clerk Model) simultaneously.
- **Location:** `utils/launch/launch-dual.sh`
- **Configuration:** Currently hardcoded to launch:
  - **Main:** Gemma-4-26B (Q4_K_XL) on Port **8085**
  - **Side:** Nemotron-3-Nano-4B (IQ4_NL) on Port **8086**

## 2. Evaluation Frameworks

### Base Framework (`utils/test/framework/`)
Evaluates a single model against a suite of general tests.
- **Launch:** `python3 utils/test/framework/main.py [model-key]`
- **Example:** `python3 utils/test/framework/main.py gemma-26b-q5`
- **Results:** Saved to `utils/test/results/framework_eval_[model]_[timestamp].md`

### Combined Framework (`utils/test/combined_framework/`)
Simulates a multi-stage workflow ("Baton Pass") using two models.
1. **Phase 1:** Clerk Model (4B) handles pre-generation tasks.
2. **Phase 2:** Main Model (26B) handles primary generation.
3. **Phase 3:** Clerk Model (4B) handles post-generation/summarization.
- **Launch:** `python3 utils/test/combined_framework/main.py`
- **Results:** Saved to `utils/test/combined_results/`

### Automated Batch Evals
- **`run_all_evals.sh`:** Iterates through all supported models, launches them one by one, and runs the coding endpoint tests.
- **`run_remaining_evals.sh`:** A helper script to resume or run specific subsets of tests.

## 3. Configuration & Model Setup

### Adding or Modifying Models
To change model paths, context sizes, or flags, edit the `models.json` file inside the respective framework directory:
- `utils/test/framework/models.json`
- `utils/test/combined_framework/models.json`

**Fields in `models.json`:**
- `path`: Absolute path to the `.gguf` file.
- `context_size`: The `-c` flag for llama-server.
- `kv_flags`: Flags for KV cache quantization (e.g., `-ctk q8_0 -ctv q8_0`).
- `gpu_layers`: Number of layers to offload to GPU (`-ngl`).

### Script Logic
- **`launch-server.sh`** uses a `case` statement to map the input argument to a specific file path and alias. If you add a new model, update the `case` block in this script.
- **Ports:** Most scripts default to **8085**. Dual-server setups use **8085** and **8086**.

## 4. Reading & Interpreting Results

Results are generated as Markdown (`.md`) files in the `results/` or `combined_results/` directories.

### Key Metrics:
- **TPS (Tokens Per Second):** The primary measure of inference speed.
- **TTFT (Time to First Token):** Measures latency/responsiveness. High TTFT usually indicates long prompt processing or system overhead.
- **Status:** Shows if the test passed or failed (e.g., due to a timeout or server crash).
- **Model Output:** The raw text generated by the model. Useful for qualitative "vibe checks" on formatting or logic.

### Troubleshooting:
- **Server Logs:** If a server fails to start, check `utils/launch/logs/`.
- **Timeouts:** If a test shows "Failed" with a timeout, check if the model was struggling with high context or if the server process died (often due to OOM/Out of Memory).
- **Ghost Processes:** If a port is blocked, run `killall -9 llama-server`. The launch scripts do this automatically, but manual intervention is sometimes needed.
